# 优化建议开发记录

## 目标
- 记录“可复用抽取点 / 性能与一致性优化点 / 工程化缺陷”等建议，便于后续分批落地。
- 本文档默认只记录建议与验收点，不直接等同于立即开发计划。

---

## 2026-03-22 v1.0

### 后端（backend）
1) 借阅逾期刷新策略
- 现状：`refreshBorrowOverdueStatuses()` 在多个接口里频繁调用，内部是 3 个 `updateMany`，数据量大时会拖慢请求。
- 优化方向：
  - 方案 A：仅在列表接口触发（`GET /api/borrows*`），写操作接口不触发；
  - 方案 B：定时任务/后台批处理（启动时 setInterval），接口读时按 `now` 计算展示态；
  - 方案 C：在写操作点精确更新单条记录的 status（减少全表更新）。
- 验收点：borrows 列表/操作延迟下降；status 变化仍符合业务预期。

2) 参数解析/校验工具抽取
- 现状：`normalizeText/toLikeRegex/parseCnPhone` 等在多个路由文件重复，容易产生“校验规则漂移”。
- 优化方向：抽到 `backend/src/utils/`（例如 `strings.ts`、`validators.ts`），统一复用。
- 验收点：重复代码减少；新增接口可直接复用公共工具。

3) 会话 last_used_at 更新
- 现状：鉴权校验命中 session 但不更新 `last_used_at`（目前仅 refresh 时更新）。
- 优化方向：在 `authMiddleware` 中对有效 session 做轻量更新（可选节流），便于审计/排查。
- 验收点：不显著增加 Mongo 压力；last_used_at 更符合实际使用。

4) 上传接口存储策略
- 现状：上传目录默认在 `process.cwd()/uploads`，缺少可观测性（空间、清理策略）。
- 优化方向：引入配置的最大文件大小限制、可选定期清理策略、或改为对象存储（仅建议，论文演示可保持本地落盘）。

### 前端（vue-vben-admin/apps/web-ele）
1) 错误提示统一
- 现状：错误 toast 对响应体字段依赖较强（优先 `error`，其次 `message`）。
- 优化方向：结合接口统一响应体后，明确只用 `message` 做提示；必要时按 `code` 分类定制文案。

2) 表格页公共能力抽取
- 现状：多页面都使用 `useVbenVxeGrid + 远程分页/排序 + 查询表单 reset`，但细节可能不一致。
- 优化方向：抽一层 `useLibraryGrid`（或同名模块）统一处理：分页/排序映射、reset 逻辑、日期范围转换等。
- 验收点：新页面接入更快；“重置仍带旧筛选值”类问题不再重复出现。

### 说明
- 上述优化建议不与本次“修 bug/统一响应体/开启自动刷新/删除用户清会话”强耦合；可单独拆小步迭代。

## 2026-03-22 v1.1

### 范围（本次落地）

#### P0（功能性 bug）
- 修复前端刷新 token / 退出登录未携带 cookie（`withCredentials` 传参错误）导致 refresh 403、自动刷新失效的问题。

#### P1（高风险一致性 / 安全）
- 收紧后端 CORS：由“回显任意 Origin + credentials”改为“可配置白名单”（开发环境默认放行本地端口）。
- 管理端 `PUT /api/users/:id/reset-password` 成功后吊销该用户所有会话（与个人中心改密策略一致）。
- 借阅相关库存回补/扣减失败时不再静默吞错：返回 500 并尽力回滚借阅记录状态，降低“记录与库存不一致”的概率。

#### P2（业务校验缺口）
- 新增/编辑图书补齐 `cover_url` 非空与格式校验（与导入规则一致：仅允许 `http(s)://` 或以 `/` 开头的相对路径）。

#### P3（复用抽取 / 工程化优化）
- 前端抽取日期范围工具（`toMs/normalizeRange`）避免多页面重复实现与规则漂移。
- Excel 导入 `importCache` 增加过期清理与容量上限，避免频繁 preview 不 commit 导致内存增长。
- 补齐 `borrows` 常用查询索引（`user_id/isbn/status/returned_at` 等），降低列表/冲突检查/上架校验的查询成本。
- 按当前页面不展示情况，移除未使用的“热门图书榜单”接口 `GET /api/analytics/hot-books` 及相关 `rank:hot_books` 逻辑与文档说明。

### 接口/页面改动点
- 前端：
  - `refreshTokenApi/logoutApi` 改为在请求配置中正确传 `withCredentials: true`。
  - 借阅管理/用户管理/我的借阅记录等页面复用同一套日期范围转换工具。
- 后端：
  - `PUT /api/users/:id/reset-password`：重置成功后删除该用户所有 sessions。
  - `POST /api/borrows/reserve`、`PUT /api/borrows/:recordId/cancel`、`POST /api/borrows/borrow`、`PUT /api/borrows/:recordId/return`：库存变更失败不再吞错，返回 500 并尽力回滚。
  - `POST /api/books`、`PUT /api/books/:isbn`：校验 `cover_url`。
  - 删除：`GET /api/analytics/hot-books`。

### 验收点
- 登录后自动刷新可用：人为将 accessToken 置为过期后能自动调用 refresh 并继续请求成功。
- 退出登录后 refresh cookie 清除，刷新接口返回 403（且前端能正确回到登录态）。
- 管理员重置某用户密码后，该用户的旧 accessToken 立刻失效（业务接口返回 401）。
- 图书新增/编辑：`cover_url` 为空或格式非法时返回 400 且有明确中文原因。
- 借阅/预约/取消/还书主链路库存一致（失败时不“假成功”）。
- Excel 导入 preview 多次后内存不持续增长（过期/超限条目会被清理）。

### 冒烟步骤（完成开发后补记录）
1) 管理员登录获取 token
2) 创建普通用户
3) 新增图书（含合法 cover_url）并上架
4) 普通用户登录并预约
5) 管理员确认借出
6) 管理员还书
7) 校验库存与记录状态
8) 管理员重置普通用户密码，校验旧 token 失效
9) 前端刷新 token / 退出登录流程走通

## 2026-03-22 v1.2

### 目标
- 提供可在 **Windows** 环境直接运行的后端冒烟脚本，用于验证 `v1.1` 的改动点（鉴权/会话、借阅库存一致性、`cover_url` 校验等）。

### 预置条件
- 后端已启动（默认 `http://localhost:3000`，可通过参数覆盖）
- MongoDB 可连接：`GET /api/health` 返回 `code=0` 且 `data.db.ok=true`
- Node.js：建议 `>=20`（Node 18+ 也可）

### 运行命令
```powershell
# 在项目根目录执行
node scripts/smoke-backend.mjs --base-url http://localhost:3000
```

### 常用参数
- `--base-url <url>`：后端地址（默认 `http://localhost:3000`）
- `--admin-username <name>`：管理员用户名（默认 `admin`）
- `--admin-password <pwd>`：管理员密码（默认 `123456`）
- `--timeout-ms <n>`：单请求超时（默认 8000）
- `--no-cleanup`：不清理本次创建的测试数据（默认会清理：测试用户/测试图书）
  - 说明：借阅记录（`borrows`）暂无删除接口，脚本会留下 2 条测试记录（取消/归还各 1 条），可按 `run_id`/用户名/书名过滤定位。

### 覆盖点（脚本会自动执行）
1) 健康检查：`GET /api/health`
2) 管理员鉴权链路：
   - `POST /api/auth/login` 获取 accessToken + refresh cookie
   - `POST /api/auth/refresh` 轮换 accessToken（校验旧 token 失效、新 token 可用）
   - `POST /api/auth/logout` 清 cookie 后 refresh 应返回 403；accessToken 也应失效
3) 用户会话吊销：
   - 管理员创建临时用户 → 用户登录
   - 管理员 `PUT /api/users/:id/reset-password`
   - 校验用户旧 accessToken 返回 401，refresh 返回 403
4) `cover_url` 校验：
   - 非法 `cover_url` 创建图书应返回 400（中文原因明确）
   - 空 `cover_url` 创建图书应自动落默认占位
5) 借阅主链路一致性（含库存）：
   - 上架 → 用户预约 → 取消预约 → 再次预约 → 管理员确认借出 → 管理员还书
   - 校验库存回到 `total_stock`
