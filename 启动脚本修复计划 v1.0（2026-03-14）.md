# 启动脚本修复计划 v1.0（2026-03-14）

## 范围
- 仅调整一键启动脚本 `start-dev.ps1` 的 **codex** 这个 tab 的启动逻辑。
- 不改动后端/前端/数据库/Redis 的启动逻辑。

## 现状与问题
- 当前 `codex` tab 启动的是 `pwsh`，再由 PowerShell 调 `wsl.exe ... bash -lc '... exec codex'`。
- 这会带来两个常见问题：
  - **tab 不是 WSL 环境**：codex 退出后会回到 PowerShell，而不是留在 WSL 里继续排查/执行命令。
  - **环境初始化不一致**：`bash -lc` 的初始化链路与“手动先进入 WSL 再运行 codex”可能不同，导致 `codex` 在交互式 WSL 中可用，但脚本里不可用（PATH/初始化脚本差异）。
  - **命令被 wt 拆分**：在 `wt` 参数中直接写 `;` 可能会被 Windows Terminal 当作 tab 命令分隔符，即使在引号里也会拆，导致后半段被当作单独命令启动而报错。

## 目标
- `codex` tab **先进入 WSL**（tab 主进程为 `wsl.exe`），再在 WSL 内：
  - 切到项目目录；
  - 启动 `codex`；
  - `codex` 退出后保留一个可交互的 WSL shell（便于继续操作）。

## 计划改动点（待确认后实施）
1. 将 `wt ... new-tab --title codex ...` 从 `pwsh -Command $codexCmd` 改为直接执行 `wsl.exe`（让 tab 处于 WSL 环境）。
2. 不做 `wslpath` 目录换算：依赖 Windows Terminal 的 `--startingDirectory $root`，让 WSL 启动时的工作目录自然落在项目目录对应的 `/mnt/<drive>/...` 路径，然后在该目录直接启动 `codex`。
3. 为 WSL 发行版提供可配置入口（优先环境变量，其次脚本参数，默认保持现状为 `Ubuntu`），降低“发行版名称不一致”导致启动失败的概率。

## 验收点
- `.\start-dev.ps1 -DryRun` 输出里，`codex` tab 的入口进程为 `wsl.exe`（而非 `pwsh`）。
- 实际运行 `.\start-dev.ps1`：
  - `codex` tab 显示为 WSL shell 环境；
  - 自动进入项目目录并启动 `codex`；
  - 结束 `codex` 后仍停留在 WSL shell（不回到 PowerShell）。

## 冒烟记录（完成改动后补充）
- 命令：
  - `.\start-dev.ps1 -DryRun`
  - `.\start-dev.ps1`
- 结果：
