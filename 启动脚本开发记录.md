- `start-dev.ps1` 中 `codex` tab 已通过 WSL 进入项目目录，但又额外使用 `codex -C <wslRoot>`，存在重复定位。

### 目标
- 仅去掉 WSL/Codex 的重复定位，保持行为不变。

### 改动点
- 保留 `wsl.exe --cd <wslRoot>`
- 将 `exec codex -C '<wslRoot>'` 简化为 `exec codex`

### 验收点
- `codex` tab 进入项目目录后直接启动 `codex`
- 不再额外使用 `-C`

### 变更记录
- 已移除 `codex -C '<wslRoot>'`
- 当时一度将 tab 标题调整为 `wsl: codex`

## 2026-03-14 v1.1

### 范围
- 仅调整 `start-dev.ps1` 中 codex tab 的启动逻辑。

### 现状问题
- 早期方案由 `pwsh` 再调 `wsl.exe ... bash -lc '... exec codex'`
- 会引出几个问题：
  - tab 退出后回到 PowerShell，而不是留在 WSL
  - `bash -lc` 初始化链路与手动进入 WSL 可能不一致
  - `wt` 参数中的 `;` 容易被错误拆分

### 目标
- 让 codex tab 先进入 WSL，再在 WSL 内启动 `codex`
- `codex` 退出后保留一个可交互的 WSL shell

### 当时方案记录
- 计划改为直接以 `wsl.exe` 作为 tab 入口
- 依赖 `--startingDirectory` 落到项目目录
- 为 WSL 发行版预留可配置能力

### 冒烟记录
- `.\start-dev.ps1 -DryRun`
- `.\start-dev.ps1`
- 当时验证结果：
  - 可进入项目目录并启动 `codex`
  - `codex` 退出后保留交互式 WSL shell

### 备注
- 本方案是阶段性方案记录；后续被 `2026-03-15 v1.2` 的“PowerShell 入口 + 进入 WSL”方案覆盖。

## 2026-03-15 v1.2

### 范围
- 仅调整 codex tab 的启动方式与外观一致性。

### 现状问题
- 直接以 `wsl.exe` 作为 tab 入口时，Windows Terminal 会套用默认 WSL profile，导致字体/字号与其它 PowerShell tab 不一致。

### 目标
- 保持 codex tab 外观与 PowerShell tab 一致
- 同时仍在 WSL 中启动 `codex`
- `codex` 退出后仍保留可交互 WSL shell；退出 WSL 后回到 PowerShell

### 最终方案
- `wt new-tab ... pwsh -NoExit -Command <codexCmd>`
- `codexCmd` 内执行：
  - `wsl.exe -d Ubuntu -- bash -ilc '(codex || true) && exec bash -il'`

### 验收点
- 运行 `.\start-dev.ps1`
- codex tab 字体/字号与其它 PowerShell tab 一致
- 自动进入 WSL 并启动 `codex`
- `codex` 退出后仍停留在 WSL shell

### 冒烟记录
- 临时验证脚本：`.\test-codex-tab.ps1`
- 结果：
  - codex 可在 WSL 中正常启动
  - tab 外观与 PowerShell tab 保持一致
  - 不再出现 `wt -l` 参数报错弹窗
