## 2026-03-22 v1.0

### 背景
- 当前仓库存在两套“通用响应体”描述：
  - 契约/实现与前端错误提示：更接近 `{ code, data, error, message }`
  - `backend/AGENTS.md`：仍写 `{ code, message, data }` 且失败 `code=HTTP状态码`
- 为减少长期维护成本，本次统一采用：`{ code, message, data }`。

### 统一约定（本次确认版）

#### 1) 通用响应体（绝大多数 JSON 接口）
```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

#### 2) 成功/失败语义
- 成功：`code === 0`，`message` 默认为 `ok`，`data` 为业务数据。
- 失败：保持正确的 HTTP 状态码（400/401/403/404/409/429/500...），同时响应体：
  - `code = HTTP 状态码`
  - `message = 中文错误原因（用于前端 toast）`
  - `data = null`

#### 3) 例外：刷新 token
- `POST /api/auth/refresh`：响应体保持“纯文本 token 字符串”（不走通用结构），前端已使用 `baseRequestClient` 调用。

### 前端对接点（结论）
- `requestClient` 解包只依赖 `code`/`data`（`code===0` 返回 `data`）。
- token 刷新触发只依赖 HTTP `401`（不依赖响应体结构）。
- 错误 toast 会从 `response.data.message` 取中文原因（无需再依赖 `error` 字段）。

### 影响范围
- 后端：需要同步修改 `ok(...)` 与统一错误处理中间件/404 兜底返回体，移除 `error` 字段。
- 文档：需要同步修改 `backend/API_CONTRACT.md` 与根目录/后端的 `AGENTS.md` 中对响应体的描述。

### 验收点
- 任意成功接口返回 `{ code:0, message:'ok', data:<payload> }`，前端页面功能无回归。
- 任意失败接口（例如无库存 409）：
  - HTTP `409`
  - body 的 `code=409`，`message` 为中文原因，前端 toast 展示该中文原因。
- `POST /api/auth/refresh` 仍返回纯文本 token，401 时能触发自动 refresh 并重放请求（开启开关后）。
