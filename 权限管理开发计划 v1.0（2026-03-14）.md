# 权限管理开发计划 v1.0（2026-03-14）

## 1. 范围
### 1.1 目标
- 实现“普通用户（读者）/管理员”两类权限的页面与接口隔离：
  - 普通用户仅展示：`图书预定`、`借阅记录`、`个人中心`。
  - 管理员展示：`数据分析`、`图书管理`、`借阅管理`、`用户管理`（维持现状）。
- 前端基于路由 `meta.authority` 做菜单/路由过滤（前端权限）。
- 后端增加最小必要的服务端鉴权（硬权限）：普通用户无法通过手输 URL/抓包请求访问管理员接口；且用户数据只能看/改自己的。

### 1.2 不做
- 不做“防抓包/防调试”类对抗（浏览器端无法真正禁止抓包，只做服务端权限校验即可）。
- 暂不细分 `admin` 与 `super` 的能力差异（本期视为同一类管理员能力）。

## 2. 现状核对（以仓库为准）
### 2.1 角色
- 后端用户表 `UserDoc.role` 当前仅支持：`'admin' | 'user'`（`backend/src/db/collections.ts`）。
- 前端 `GET /api/user/info` 返回的 `roles` 当前映射为：
  - `user.role === 'admin'` -> `roles: ['super']`
  - `user.role === 'user'` -> `roles: ['user']`
  - 结论：前端权限过滤应以 `super/user` 作为 authority 值（本期先沿用现有映射，避免破坏现有登录态逻辑）。

### 2.2 借阅/预约状态（BorrowStatus）
后端 `BorrowStatus` 当前包含：
- `borrowed`（借阅中）
- `returned`（已归还）
- `overdue`（逾期：可能是“逾期未还”，也可能是“预约超时未取”）
- `reserved`（已预约/待取书）
- `canceled`（已取消）

## 3. 接口/后端改动点（backend）
### 3.1 通用鉴权规则（硬权限）
- 继续沿用 `authMiddleware` 解析 `Authorization: Bearer <token>` 并写入 `ctx.state.auth`。
- 新增/补齐按角色限制的检查：
  - 管理员专用接口：普通用户调用返回 `403 Forbidden`。
  - 用户专用接口：只能操作自己的数据（基于 `ctx.state.auth.userId/username` 强制约束，忽略前端传入的 `username` 等敏感筛选条件）。
  - 说明：前端“隐藏菜单/路由”只是 UX；真正的权限边界以服务端校验为准。

### 3.2 管理员接口限制（建议）
- 仅管理员可访问：
  - 用户管理：`/api/users*`、`/api/users-import*`
  - 数据分析：`/api/analytics*`
  - 图书导入：`/api/books-import*`
  - 图书维护写接口：`POST/PUT /api/books*`、`PUT /api/books/:isbn/shelf`
  - 借阅管理相关“代用户借书/还书”：现有 `POST /api/borrows/borrow`、`PUT /api/borrows/:recordId/return` 维持管理员操作语义
    - 备注：当前 `POST /api/borrows/borrow` 允许传入 `username`，若不限制将导致普通用户可替别人借书（必须收口）。

### 3.3 用户端能力补齐（预约/记录）
为满足“线上预定、线下取书、未取为待取书”的论文逻辑，增加用户端接口（最小闭环）：
- `GET /api/borrows/my`：查询“我的借阅/预约记录”
  - 后端强制 `user_id = ctx.state.auth.userId`（普通用户不允许查询他人）。
- `POST /api/borrows/reserve`：用户预约图书
  - 入参：`isbn`
  - 行为：创建一条 `BorrowDoc.status='reserved'` 的记录，并扣减库存（`current_stock - 1`）
  - `due_date` 作为“取书截止时间”（例如默认 3 天）；超时后前端可显示为 `overdue`（由现有 effective status 逻辑计算）。
- `PUT /api/borrows/:recordId/cancel`：用户取消预约（仅限自己的 `reserved` 记录）
  - 行为：记录置为 `canceled` 并回滚库存（`current_stock + 1`，不超过 `total_stock`）。
- （可选，管理员线下确认取书）`PUT /api/borrows/:recordId/confirm-borrow`
  - 将 `reserved` 记录转为 `borrowed`，并设置新的借阅 `due_date`（例如 30 天）。
  - 说明：如果暂不做此接口，论文口径可先表述为“管理员在借阅管理中为用户办理借阅（生成借阅记录）”，但会出现“预约记录与借阅记录割裂”的体验。

## 4. 页面/前端改动点（vue-vben-admin）
### 4.1 菜单/路由权限（前端）
- 在路由模块（`vue-vben-admin/apps/web-ele/src/router/routes/modules/library.ts`）为每个一级路由增加 `meta.authority`：
  - 管理员页面：`authority: ['super']`
  - 普通用户页面：`authority: ['user']`
- 同步调整后端 `GET /api/user/info` 的 `homePath`：
  - `super` -> `/analytics`
  - `user` -> `/user-reservations`
- 开发期测试开关（便于联调，但默认必须关闭）：
  - 仅当 `VITE_ENABLE_ADMIN_USER_PAGES=true` 时，才允许“管理员看到并访问用户端页面”（用户端 authority 临时扩展为 `['user','super']`）。
  - 注意：不要在 `DEV` 环境默认开启，否则会导致管理员侧边栏出现 `图书预定/借阅记录/个人中心` 等用户端菜单（你截图里的问题即由此触发）。

### 4.1.1 后端菜单（backend accessMode）
- `GET /api/menu/all` 按 `ctx.state.auth.role` 返回不同菜单树：
  - 管理员：管理端 4 个菜单
  - 普通用户：用户端菜单
  - 备注：该接口属于“菜单数据”，不替代服务端接口鉴权。

### 4.2 新增用户端页面（建议新开，避免与管理员页混杂）
- `图书预定`：新页面
  - 路由：`/user-reservations`
  - 内容：展示可预约图书列表 + “预约”按钮
- `借阅记录`：新页面
  - 路由：`/user-borrow-records`
  - 内容：展示我的记录列表；对 `reserved` 提供“取消预约”
- `个人中心`：复用现有 `/profile` 页面（不动）
  - 备注（开发期测试开关）：
    - 默认：用户端页面仅对 `authority: ['user']` 可见（避免管理员菜单割裂）。
    - 开发期：为方便联调，可通过环境变量显式开启 `VITE_ENABLE_ADMIN_USER_PAGES=true`，让管理员也能看到并访问用户端页面（将用户端页面 authority 临时扩展为 `['user', 'super']`）。
    - 收尾：提交/答辩前关闭开关（或移除该逻辑），不影响既有管理员后台功能。

## 5. 验收点
### 5.1 普通用户（roles=['user']）
- 左侧菜单仅出现：`图书预定`、`借阅记录`（`个人中心`在顶部入口可进入）。
- 访问 `/analytics`、`/users`、`/borrows(管理员页)` 等路由时：路由不注册或跳 403（取决于前端策略）。
- 抓包/手输 URL 访问管理员接口（如 `/api/users`、`/api/analytics`）返回 403。
- 预约一本有库存的书：库存扣减；在“借阅记录”看到状态 `reserved/待取书`。
- 取消预约：记录变为 `canceled`；库存回滚。

### 5.2 管理员（roles=['super']）
- 左侧菜单维持现状（数据分析/图书管理/借阅管理/用户管理）。
- 左侧菜单：
  - 默认不出现：`图书预定`、`借阅记录`；
  - 若开启“开发期测试开关”，则允许出现并可访问（便于联调）。
- 管理员可正常进行借阅管理（代借/还书）与用户管理操作。
