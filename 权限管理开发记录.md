## 2026-03-14 v1.0

### 目标
- 实现普通用户与管理员的页面、菜单、接口隔离
- 前端基于 `meta.authority` 做路由过滤
- 后端补齐最小必要的服务端硬权限

### 不做
- 不做浏览器端“防抓包/防调试”类对抗
- 暂不细分 `admin` 与 `super` 的能力差异

### 现状结论
- 后端角色：`admin | user`
- 前端 authority 映射：
  - `admin -> ['super']`
  - `user -> ['user']`
- 借阅/预约状态包含：
  - `borrowed`

## 2026-03-15 v1.1

### 目标
- 将当前“后端两角色 + 前端伪映射”的权限实现，收敛为真实三角色模型
- 明确用户管理中的角色边界与可编辑范围

### 当前现状
- 当前后端真实角色仍为：
  - `admin`
  - `user`
- 当前前端权限标识中：
  - `admin` 被映射为 `super`
  - `user` 被映射为 `user`
- 因此当前仓库尚未形成真实的 `super | admin | user` 三角色模型

### 目标角色模型
- `user`
  - 无用户管理菜单权限
  - 不可管理其他用户
  - 仅可在个人中心修改自己的资料与密码
- `admin`
  - 拥有用户管理菜单权限
  - 只能修改 `user`
  - 不可修改 `super`
  - 不可修改任何人的角色
  - 可在个人中心修改自己的资料与密码
- `super`
  - 拥有完整用户管理菜单权限
  - 可以修改 `admin` 和 `user`
  - 只有 `super` 可以改动角色
  - 可在个人中心修改自己的资料与密码

### 内置账号约束
- 不新增新的 `super` 账号
- 沿用现有内置账号 `vben`
- 后续将 `vben` 从当前普通用户调整为真实 `super`
- 默认密码仍为 `123456`

### 用户管理约束
- `user`
  - 无用户管理页面入口
- `admin`
  - 可编辑目标：`user`
  - 不可编辑目标：`admin`、`super`
  - 不可修改角色字段
- `super`
  - 可编辑目标：`admin`、`user`
  - 可修改角色字段

### 开发顺序
1. 先修正个人中心与右上角的展示问题
2. 再补真实三角色模型
3. 最后收敛用户管理中的编辑边界与角色修改能力

### 预期验收点
- `user` 登录后无用户管理菜单
- `admin` 登录后只能编辑 `user`
- `admin` 不可修改角色字段
- `super` 登录后可编辑 `admin/user`，并可修改角色字段
- 内置账号 `vben` 以真实 `super` 身份参与登录与权限校验

## 2026-03-22 v1.1

### 范围
- 前端：开启自动刷新 `accessToken`（默认开）
  - `vue-vben-admin/packages/@core/preferences/src/config.ts`
- 后端：删除用户同时清理会话
  - `backend/src/routes/users.ts`
  - `backend/src/db/auth.ts`

### 改动点
1) 自动刷新 token（默认开启）
- 将 `preferences.app.enableRefreshToken` 默认值从 `false` 调整为 `true`
- 说明：刷新逻辑仍以 HTTP `401` 为触发条件，刷新接口仍为纯文本 token（`POST /api/auth/refresh`）

2) 删除用户同时清理会话
- `DELETE /api/users/:id` 成功删除用户后，同时删除该用户的 sessions（避免脏数据与无意义 refresh 尝试）

### 验收点
- access token 过期后，前端会自动调用 `POST /api/auth/refresh` 刷新并重放原请求（无感恢复）
- 删除用户后：
  - 该用户用旧 token 访问业务接口：HTTP `401`（用户不存在/已删除）
  - 该用户 refresh：HTTP `403`（refresh token 不存在/无效）
